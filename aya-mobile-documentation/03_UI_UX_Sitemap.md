```
**ملاحظة هامة (Blind POS):** موظف الـ POS لا يرى هذه الشاشة الرئيسية ولا يرى أي إجماليات مالية (المبيعات، الصندوق، الأرباح، التكلفة). شاشته الافتراضية والوحيدة هي شاشة نقطة البيع (POS) فقط. أحمد (Admin) فقط هو من يرى المبالغ والأرباح.
```

**للـ Admin فقط:**
- إعدادات النظام (`system_settings`)
- Audit Log
- **اللقطة اليومية المزدوجة (Hybrid Daily Snapshot):** زر حفظ لقطة يدوية + تعمل تلقائياً (Cron Job) عند 12:00 منتصف الليل.
- إدارة فئات المصروفات

**عناصر إضافية في الـ Dashboard (Transparent Profit للـ Admin):**
- 📊 **ملخص الأرباح شفاف ومفصل:**
  - `(المبيعات - التكلفة = الربح التشغيلي)`
  - `(الربح التشغيلي - المصروفات = صافي الربح)`
- 🔔 أيقونة الإشعارات (مع عداد الغير مقروءة) في الشريط العلوي
- 📦 قائمة منتجات منخفضة المخزون (إذا وجدت)
- 📝 آخر 5 عمليات (فواتير، تسديدات، مرتجعات، تسويات)
- ⚠️ ديون متأخرة (إذا وجدت)

**بطاقة سلامة الأرصدة (Balance Integrity Card) — Admin فقط:**

```
┌───────────────────────────────────────┐
│  ⚕️ سلامة الأرصدة المالية                │
├───────────────────────────────────────┤
│                                         │
│  ✅ الحالة: سليمة                        │
│  آخر فحص: اليوم 10:30 صباحاً           │
│                                         │
│  ╌╌╌╌╌╌╌╌╌╌╌╌╌ أو ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌   │
│                                         │
│  🔴 الحالة: فروقات مكتشفة              │
│  آخر فحص: اليوم 10:30 صباحاً           │
│  ───────────────────────────────────  │
│  ⚠️ الصندوق: فرق -150 د.أ            │
│  ⚠️ Orange Money: فرق +30 د.أ         │
│                                         │
│  [ 🔄 إعادة الفحص ]  [ 📄 التفاصيل ]  │
│                                         │
└───────────────────────────────────────┘
```

**الأزرار:**
- **إعادة الفحص** → `POST /api/health/balance-check` → يستدعي `fn_verify_balance_integrity()` ويعرض النتيجة
- **التفاصيل** → يفتح نافذة بجدول تفصيلي: اسم الحساب | المتوقع | الفعلي | الفرق

**الحالات:**
| الحالة | الأيقونة | اللون | المعنى |
|--------|---------|-------|--------|
| `ok` | ✅ | أخضر | الأرصدة متطابقة |
| `drift` | 🔴 | أحمر | فروقات مكتشفة — تحقق فوراً |
| لم يُفحص | ❓ | رمادي | لم يُجرك الفحص بعد |

> **API Route:** `POST /api/health/balance-check`
> - **الصلاحية:** Admin فقط
> - **الدالة:** `fn_verify_balance_integrity()`
> - **الاستجابة (ok):** `{ success: true, data: { status: "ok", accounts: [...], checked_at: "..." } }`
> - **الاستجابة (drift):** `{ success: true, data: { status: "drift", drifts: [{ account_name, expected, actual, diff }], checked_at: "..." } }`
> - **التنبيه اليومي (V1):** Cron Job يستدعي نفس الـ Route → إذا `drift` يُنشئ إشعار في `notifications` مع `type: reconciliation_difference`

## 🎨 مبادئ التصميم UX

### 1. السرعة في POS
- البحث السريع (أقل من ثانية)
- اختصارات لوحة المفاتيح
- أزرار كبيرة للمس
- لا حاجة للتمرير في الشاشة الرئيسية

### 2. وضوح المعلومات
- ألوان للحالات (أخضر/أحمر/أصفر)
- أرقام كبيرة للمبالغ
- تنبيهات واضحة للأخطاء
- تأكيد قبل العمليات الحساسة

### 3. تقليل الأخطاء
- تأكيد قبل الحذف
- التحقق من البيانات قبل الحفظ
- إلغاء متاح في أي خطوة
- لا يوجد تراجع تلقائي بعد الحفظ؛ التصحيح يتم عبر قيد عكسي + Audit Log

### 4. التوافق مع الأجهزة
- تصميم متجاوب (Responsive) **إلزامي** لكل الشاشات بدون استثناء.
- يعمل على موبايل (360px+) مع أزرار لمس كبيرة وسلوك عمودي افتراضي.
- يعمل على تابلت (768px+) مع توزيع هجين (قائمة/محتوى) حسب الاتجاه.
- يعمل على لابتوب (1024px+) مع تخطيط كامل واختصارات لوحة مفاتيح.
- لا يُسمح بوجود شاشة "Desktop Only" في MVP أو V1.
- نفس رابط النظام يعمل على كل الأجهزة (Web App)، مع دعم التثبيت على الجهاز (A2HS/Install App) للوصول السريع.
- دعم التثبيت لا يعني وضع Offline: عند انقطاع الشبكة تطبق `ERR_NETWORK` كالمعتاد.

**معيار قبول تصميمي مختصر:**
1. لا يوجد قص أفقي (Horizontal Overflow) عند `360px`.
2. كل عناصر التفاعل الأساسية قابلة للاستخدام باللمس على الهاتف.
3. شاشة POS تعمل باللمس على الهاتف/التابلت وبالكيبورد على اللابتوب.

### قواعد التصدير

| القاعدة | التفاصيل |
|---------|---------|
| **الصلاحية** | POS لا يصدّر إلا تقارير المبيعات اليومية وإيصالات الفواتير |

| **الحجم** | التصدير محدود بـ 10,000 سجل لكل ملف — للأكبر: فلتر بالتاريخ |
| **الأسماء** | أسماء الملفات: `{report_type}_{date_range}_{timestamp}.csv` |
| **اللغة** | عناوين الأعمدة بالعربية |

## 🚀 إضافات V2 المخططة (Post-PX-07 Addendum)

> هذا القسم يحدد الأسطح التي ستُفتح في `PX-08 .. PX-14`.
> شاشات المصروفات والإشعارات المعرّفة أعلاه تصبح **first-class operational surfaces** في `PX-08`.

### شاشة عامة: عرض الإيصال عبر رابط (`/r/[token]`)

| البند | التفاصيل |
|-------|----------|
| **الهدف** | مشاركة إيصال فاتورة للقراءة فقط خارج النظام |
| **المستخدمون** | عام عبر token opaque فقط |
| **الصلاحية** | لا تحتاج تسجيل دخول لكن تحتاج token صالح غير ملغى وغير منتهي |

**المحتوى المسموح فقط:**
- اسم المتجر
- رقم الفاتورة
- التاريخ
- عناصر الفاتورة
- الإجمالي
- وسائل الدفع المسجلة على مستوى الفاتورة فقط

**المحتوى المحجوب إلزاميًا:**
- `cost_price`
- `profit`
- `avg_cost_price`
- `internal_notes`
- `audit data`
- `current_balance`
- أي بيانات إدارية داخلية

**الملاحظات:**
- الرابط يدعم `share` و`window.print()`
- واجهة mobile-first
- في حال token غير صالح/ملغى/منتهي: شاشة خطأ صريحة بلا أي تسريب معلومات

## 📱 حالات الشاشة (Screen States)

### Loading State
- عرض Skeleton placeholder أثناء تحميل البيانات
- لا تُعطّل التفاعل مع العناصر المُحمّلة

### Empty State
- رسالة واضحة: "لا توجد بيانات"
- زر إجراء مناسب (مثل: "+ إضافة أول منتج")

### Error State
- رسالة خطأ بالعربية من [16_Error_Codes.md](./16_Error_Codes.md)
- زر "إعادة المحاولة"
- لا تُفقد البيانات المُدخلة

### Offline State (حالة انقطاع الشبكة)
- Banner علوي: "⚠️ لا يوجد اتصال"
- تعطيل أزرار الحفظ
- إبقاء البيانات المعروضة
