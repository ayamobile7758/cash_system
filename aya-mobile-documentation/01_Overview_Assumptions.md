## 👥 المستخدمون والصلاحيات

### 1) أحمد (Admin)
- صلاحيات كاملة على جميع الوظائف
- إدارة المستخدمين والمنتجات والموردين
- إدارة عملاء الدين وتحديد الحدود
- التقارير الشاملة
- إدارة الإعدادات العامة (مثل حد الخصم والعمولات)
- اعتماد ومراجعة التنبيهات التشغيلية والمالية

### 2) موظف المبيعات (POS)
- نقطة البيع والعمليات اليومية
- عرض الفواتير والمرتجعات والتقارير اليومية
- يمكنه: تسجيل مرتجع، تسوية حساب، تحويل مالي
- لا يمكنه: إضافة منتجات، إدارة موردين، إضافة عميل دين، تسجيل دين يدوي

## ⚠️ الافتراضات العملية (Assumptions)

### افتراضات أساسية
1. **لا يوجد باركود:** جميع المنتجات تُختار يدوياً من القائمة
2. **لا IMEI/Serial:** لا نتتبع أرقام الجهاز الفردية
3. **سعر بيع واحد:** لا يوجد أسعار متعددة للمنتج نفسه
4. **لا تتبع أرقام الشحن:** نسجل المبلغ والربح فقط
5. **الخصم على مستوى المنتج:** يُطبق خصم لكل منتج على حدة (وليس على الفاتورة ككل). موظف POS: حتى 10%. أحمد: بلا حد. أي خصم يُسجل مع `discount_by` في Audit Log (متوافق مع ADR-016)
6. **المرتجعات بدون قيود زمنية:** يمكن الإرجاع في أي وقت
7. **إعادة المبلغ بأي طريقة:** ليس شرطاً نفس طريقة الدفع الأصلية

### افتراضات مالية
8. **العملة:** الدينار الأردني (JOD)
9. **الفيزا عمولة:** مصروف منفصل وقابل للتعديل لكل حساب فيزا من الإعدادات
10. **آخر تكلفة + متوسط تكلفة:** نحتفظ بالاثنين للمنتج
11. **دفعات الموردين:** مدعومة في V1 (المورد اختياري بالنقدي، وإلزامي بالشراء الآجل والتسديد)
12. **حد الدين:** تنبيهي فقط، لا يمنع الإتمام (قيمة افتراضية 10 د.أ قابلة للتعديل من الإعدادات)
13. **استحقاق الدين:** 30 يوم افتراضياً

### افتراضات تشغيلية
14. **الجرد:** يومي/أسبوعي/شهري مع جلسات جرد. عند إكمال الجرد يُحدّث `products.stock_quantity` بالكمية الفعلية من `inventory_count_items.actual_quantity`
14b. **تسديد الدين FIFO:** كل دفعة تسديد تُطبّق أولاً على الدين الأقدم (`MIN(due_date)`) حتى يصفر ثم التالي. أحمد يستطيع تجاوز FIFO وتحديد الدين يدوياً من الواجهة (V1)
15. **لا يوجد إقفال يومي:** الاعتماد على التقارير بالتاريخ + Daily Snapshot عند الطلب
16. **التسويات:** Actual vs Expected مع تسجيل الفرق
17. **Audit Log:** تسجيل جميع العمليات الحساسة
18. **تشغيل متزامن:** يدعم أكثر من نقطة بيع في نفس الوقت داخل نفس المتجر

### افتراضات تقنية
19. **الوضع Offline:** لا يوجد دعم offline نهائياً (في جميع الإصدارات)
20. **معمارية API-First (ADR-042):** جميع عمليات الكتابة تمر عبر Next.js API Routes — الواجهة لا تُدرج بيانات مباشرة في قاعدة البيانات
21. **الاحتكار الحسابي للسيرفر (ADR-043):** الأسعار والإجماليات تُحسب في الباك-إند حصرياً — الواجهة تُرسل نية الإجراء فقط (product_id + quantity)
22. **نموذج Revoke-All-First (ADR-044):** صلاحيات الكتابة مسحوبة افتراضياً من الواجهة — القراءة فقط عبر RLS
23. **الطباعة:** غير متاحة حالياً (مؤجلة)
24. **الإشعارات:** إشعارات داخلية في جميع المراحل
25. **واتساب:** `wa.me` deep link فقط — النظام يجهز الرسالة، المستخدم يضغط إرسال يدوياً. لا API خارجي. يوجد `delivery log` محلي Auditable بحالة baseline (`queued`) دون تأكيد تسليم خارجي ودون تخزين raw phone.
26. **النسخ الاحتياطي:** زر تحميل محلي يُصدّر جميع بيانات المستخدم (JSON) — لا Supabase Scheduled Backup
27. **المنطقة الزمنية:** Asia/Amman (الأردن)
28. **المتصفحات المدعومة:** Chrome/Edge/Safari/Firefox (آخر إصدارين) مع دعم Web APIs القياسية
29. **الأجهزة:** هاتف + تابلت + لابتوب + ديسكتوب — الواجهة Responsive تعمل من `360px+`
30. **المرجع المعتمد للعقد:** [29_Device_Browser_Policy.md](./29_Device_Browser_Policy.md)
31. **منصة الاستضافة:** Supabase Free Tier + Vercel Hobby (مجاني) — القيود: 500MB DB, 50K Active Users/شهر
32. **المطور:** AI Agent — التوثيق مُصمَّم ليكون مرجعاً كافياً للتنفيذ دون أسئلة إضافية

## 🎯 أهداف النظام

### الأهداف الرئيسية
1. تسريع عملية البيع في نقطة البيع (POS)
2. منع الأخطاء والتلاعب عبر قيود قاعدة البيانات
3. تتبع الديون والتنبيه بها
4. إدارة المخزون بدقة
5. توليد تقارير مالية واضحة
6. تسهيل المرتجعات والاستبدال
7. تتبع يومي دقيق عبر فلاتر التاريخ وDaily Snapshot
8. عرض هيستوري مبيعات واضح وقابل للبحث
9. دعم البيع المتزامن من جهازين+ بدون تعارض مخزون

### معايير النجاح
- وقت البيع أقل من 2 دقيقة
- خطأ في الفاتورة أقل من 1%
- تتبع 100% للديون
- جرد دقيق بدون فروقات غير مفسرة

## 🔴 الثغرات والمخاطر والتوصيات (Gaps, Risks & Recommendations)

### المخاطر المالية الحرجة

#### 1. المرتجعات بأي طريقة دفع
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | إعادة المبلغ بأي طريقة دفع قد تسبب فروقات في الحسابات |
| **السيناريو** | عميل دفع نقدي، يُرجع ويُعاد له بالفيزا → الفيزا عمولة تُخصم مرتين |
| **الخطورة** | عالية جداً |

**التوصية:**
- تسجيل طريقة الإرجاع منفصلة عن طريقة الدفع الأصلية
- حساب الفرق في العمولة وتسجيله كـ "مصروف استرداد"
- ربط المرتجع بالفاتورة الأصلية لحساب الفروقات

---

#### 2. الدفع المختلط (Split Payment)
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | تجزئة الدفع قد تؤدي لعدم توازن الحسابات |
| **السيناريو** | نقدي 100 + فيزا 200 = 300، لكن الفيزا عمولة 2% = 196 فعلياً |
| **الخطورة** | عالية |

**التوصية:**
- تسجيل كل جزء كـ Ledger Entry منفصل
- حساب صافي كل طريقة بعد العمولة
- التحقق من أن مجموع الصافي = صافي الفاتورة

---

#### 3. الديون داخل الفاتورة + الديون اليدوية
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | تكرار تسجيل الدين أو تجاوز الحد |
| **السيناريو** | عميل له حد 100، فاتورة 80 دين + دين يدوي 30 = 110 |
| **الخطورة** | عالية جداً |

**التوصية:**
- دالة DB موحدة `check_debt_limit(customer_id, amount)`
- حساب الرصيد الحقيقي: فواتير الدين - تسديدات + ديون يدوية
- تسجيل تنبيه تلقائي عند التجاوز + إشعار أحمد (بدون منع العملية)

---

#### 4. عمولة الفيزا
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | أين تُسجل العمولة؟ كخصم من المبلغ أم كمصروف منفصل؟ |
| **السيناريو** | فيزا 1000، عمولة 2% = 20، هل نسجل 1000 ومصروف 20 أم 980 فقط؟ |
| **الخطورة** | متوسطة |

**التوصية:**
- تسجيل المبلغ الكامل 1000 كـ "وارد"
- تسجيل 20 كـ "مصروف عمولة فيزا"
- هذا يوضح الإيراد الحقيقي والتكلفة

---

#### 5. التسويات (Reconciliation)
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | الفرق بين المتوقع والفعلي قد يكون سرقة أو خطأ |
| **السيناريو** | الصندوق المتوقع 5000، الفعلي 4800 → الفرق 200 |
| **الخطورة** | عالية |

**التوصية:**
- تسجيل الفرق كـ "فرق تسوية" مع سبب إلزامي
- إشعار أحمد فوراً بالفروقات
- عدم اعتماد Daily Snapshot قبل توضيح الفروقات

---

#### 6. عدم السماح بالعمليات بأثر رجعي (No Backdating — ADR-034 مُعدّل)
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | إدراج قيود أو فواتير بتاريخ سابق يكسر اللقطات اليومية ويسبب انحراف الأرصدة (Balance Drift) |
| **السيناريو** | حاول موظف أو مدير تسجيل عملية بتاريخ أمس |
| **الخطورة** | عالية جداً |

**التوصية (مُعتمدة):**
- **جميع العمليات تُسجل بـ `CURRENT_DATE` فقط — لا استثناءات حتى للـ Admin**
- أي تصحيح لخطأ سابق يتم عبر إدخال **قيد تسوية (adjustment)** بتاريخ اليوم الحالي مع وصف نصي يشير للمرجع الأصلي
- Audit Log إلزامي لكل عملية تسوية
- Daily Snapshot عند الطلب لحفظ مرجع يومي

---

### المخاطر التشغيلية

#### 7. الخصم بدون رقابة
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | الخصم المتاح للجميع قد يُسيء استخدامه |
| **السيناريو** | موظف يُعطي خصومات كبيرة لأصدقائه |
| **الخطورة** | متوسطة |

**التوصية:**
- تسجيل كل خصم مع من نفذه
- تقرير يومي للخصومات لأحمد
- حد خصم تنبيهي (افتراضي 10%) قابل للتعديل من الإعدادات + إشعار أحمد عند تجاوزه

---

#### 8. المرتجعات بدون قيود زمنية
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | إرجاع منتج بعد فترة طويلة قد يكون سعره تغير |
| **السيناريو** | شراء بـ 5000، إرجاع بعد شهر والسعر الآن 4500 |
| **الخطورة** | متوسطة |

**التوصية:**
- إرجاع بنفس سعر الفاتورة الأصلية
- تسجيل سبب الإرجاع إلزامياً
- إشعار أحمد بالمرتجعات الكبيرة

---

#### 9. الجرد (Inventory Count)
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | الفروقات قد تكون سرقة أو تلف أو خطأ |
| **السيناريو** | جرد يومي يظهر نقص 5 قطع بدون سبب |
| **الخطورة** | متوسطة |

**التوصية:**
- تسجيل سبب الفرق إلزامياً
- خيارات: سرقة، تلف، خطأ إدخال، بيع غير مسجل
- تقرير الفروقات لأحمد

---

#### 10. عملاء الدين
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | إضافة عميل دين قد يكون خطأً |
| **السيناريو** | موظف يضيف نفسه كعميل دين |
| **الخطورة** | عالية |

**التوصية:**
- إضافة عملاء الدين لأحمد فقط
- توثيق بيانات العميل (هاتف + رقم هوية اختياري عند الحاجة)
- حد أقصى لكل عميل

---

#### 11. المرتجع بعد تسديد دين جزئي
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | عميل اشترى بالدين، سدد جزءاً، ثم أرجع البضاعة — كيف يُعالج؟ |
| **السيناريو** | فاتورة 1000 دين → سدد 400 → أرجع البضاعة. هل يسترد 1000 أم 600؟ |
| **الخطورة** | عالية |

**التوصية:**
- المرتجع يُرجع **سعر البضاعة الأصلي** (1000) كقيد عكسي
- يُلغى قيد الدين المرتبط بالفاتورة
- المبلغ المسدد سابقاً (400) يُعاد للعميل بالطريقة التي يختارها
- الصافي: العميل يسترد 400 نقداً/فيزا + يُلغى الدين المتبقي 600
- تسجيل كامل في Audit Log مع ربط بالفاتورة الأصلية وقيد الدين

---

#### 12. الدفع الزائد (Overpayment)
| الجانب | الخطر |
|--------|-------|
| **المشكلة** | العميل يدفع مبلغاً أكبر من المطلوب |
| **السيناريو** | فاتورة 950، العميل يدفع 1000 |
| **الخطورة** | منخفضة |

**التوصية:**
- النظام يحسب **الباقي** (الفرق) ويعرضه للموظف: "الباقي: 50 د.أ"
- الباقي يُسلّم للعميل يدوياً (لا يُسجل كرصيد دائن في النظام)
- الفاتورة تُسجل بالقيمة الفعلية (950) وليس بالمبلغ المدفوع (1000)
- لا يوجد مفهوم "رصيد دائن" أو "محفظة عميل" في هذا الإصدار
- في حالة تسديد دين: لا يُسمح بتسديد أكثر من المتبقي (`ERR_DEBT_OVERPAY`)

### 🛡️ توصيات التصميم الوقائية

#### 1. قيود قاعدة البيانات (Constraints)

```
✅ تنفيذ في DB (بدون كود في Frontend)
```

| القيد | الجدول | الغرض |
|-------|--------|-------|
| `CHECK (amount >= 0)` | جميع الجداول المالية | منع الأرقام السالبة |
| `CHECK (discount <= subtotal)` | invoices | منع خصم غير منطقي |
| `UNIQUE (invoice_number)` | invoices | منع تكرار أرقام الفواتير |
| `FOREIGN KEY` بسياسة حذف واضحة لكل علاقة | جميع العلاقات | منع تلف الترابط المرجعي |
| `CHECK (quantity > 0)` | invoice_items | منع صفوف فارغة |
| `CHECK (payment_date <= CURRENT_DATE)` | debt_payments | منع إدخال تاريخ مستقبلي |

#### 1.1 سياسة الخصومات (ADR-016)

| القاعدة | التفاصيل |
|---------|----------|
| **موظف POS** | حتى 10% من قيمة المنتج الواحد |
| **أحمد (Admin)** | بلا حد |
| **التنبيه** | تجاوز الحد ← إشعار داخلي لأحمد |
| **التسجيل** | `discount_by` في الفاتورة + Audit Log |
| **الإعداد** | `system_settings.max_pos_discount_percentage = 10` |

---

#### 1.2 سياسة إلغاء الفاتورة (ADR-017)

| القاعدة | التفاصيل |
|---------|----------|
| **مَن يلغي** | أحمد (Admin) فقط |
| **السبب** | إلزامي (`cancel_reason`) |
| **الشرط** | لا يوجد مرتجع مرتبط بالفاتورة |
| **الأثر** | عكس المخزون + عكس Ledger + إلغاء الدين إن وجد |
| **التسجيل** | `cancelled_by` + Audit Log + إشعار |

---

#### 1.3 سياسة حذف البيانات (ADR-022)

| الكيان | سياسة الحذف | السبب |
|--------|-------------|-------|
| منتجات | Soft Delete (`is_active = false`) | مرتبطة بفواتير سابقة |
| عملاء دين | Soft Delete | مرتبطون بقيود مالية |
| موردين | Soft Delete | مرتبطون بمشتريات |
| حسابات | Soft Delete | مرتبطة بـ Ledger Entries |
| فواتير | لا تُحذف أبداً | سجل مالي دائم |
| Ledger Entries | لا تُحذف أبداً | القيود المحاسبية لا تُمسح |

#### 1.4 التزامن والأخطاء (ADR-023)

| القاعدة | التفاصيل |
|---------|----------|
| **Transactions** | كل عملية معقدة = PostgreSQL Transaction |
| **Locking** | `SELECT FOR UPDATE` على المخزون والأرصدة |
| **Rollback** | فشل أي خطوة يلغي العملية بالكامل |
| **رسالة الخطأ** | واضحة بالعربية (مثل: "المخزون غير كافٍ") |

#### 4. Audit Log (سجل التدقيق)

| العملية | ما يُسجل |
|---------|----------|
| إنشاء فاتورة | رقم الفاتورة، المبلغ، المستخدم، الوقت |
| إلغاء فاتورة | رقم الفاتورة، السبب، المستخدم، القيم قبل/بعد |
| تعديل فاتورة | الفرق، السبب، المستخدم |
| خصم | المبلغ، النسبة، من نفذه، الفاتورة |
| خصم يتجاوز الحد | نفس الخصم + تنبيه |
| مرتجع | رقم المرتجع، الفاتورة الأصلية، السبب |
| تسديد دين | المبلغ، العميل، المستخدم |
| تجاوز حد دين | العميل، الحد، الرصيد الجديد |
| تسوية | الفرق، السبب، الحساب |
| إنشاء Daily Snapshot | التاريخ، المستخدم، الملخص |
| محاولة Backdating مرفوضة | نوع العملية، التاريخ المرسل، المستخدم، سبب الرفض |
| تعطيل/تفعيل كيان | الجدول، الكيان، المستخدم |

---

#### 5. Locks (أقفال النظام)

| القفل | متى | من يفكه |
|-------|-----|---------|
| فاتورة مُرجعة | بعد إرجاع الفاتورة | أحمد فقط |
| عميل دين متجاوز | عند تجاوز الحد | لا يوجد قفل (تنبيه فقط) |
| جرد جاري | أثناء الجرد | أي مستخدم |

### 📊 مصفوفة المخاطر

| المخطر | الاحتمالية | التأثير | المستوى | المعالجة |
|--------|-----------|---------|---------|----------|
| محاولة تسجيل تاريخ سابق (Backdating Attempt) | متوسط | عالي جداً | 🔴 حرج | منع عبر CHECK + Audit Log + إشعار |
| تجاوز حد الدين | متوسط | عالي | 🔴 حرج | تنبيه فقط (ADR-006) |
| تعارض مستخدمَين على نفس المنتج | منخفض | عالي | 🔴 حرج | SELECT FOR UPDATE (ADR-023) |
| فروقات تسوية غير موضحة | عالي | متوسط | 🟠 عالي | سبب إلزامي + إشعار |
| خصم مفرط | عالي | متوسط | 🟠 عالي | حد 10% للموظف (ADR-016) |
| إلغاء فاتورة بدون سبب | منخفض | عالي | 🟠 عالي | Admin فقط + سبب إلزامي (ADR-017) |
| مرتجع بدون ربط بالأصل | منخفض | عالي | 🟠 عالي | FK إلزامي |
| عمولة فيزا غير محسوبة | متوسط | متوسط | 🟡 متوسط | حساب تلقائي |
| خطأ في الجرد | عالي | منخفض | 🟡 متوسط | سبب إلزامي |
| دين يدوي مكرر | منخفض | متوسط | 🟡 متوسط | Audit Log |
| حذف بيانات مرتبطة | منخفض | عالي | 🟡 متوسط | Soft Delete (ADR-022) |
| مرتجع بعد تسديد جزئي | منخفض | عالي | 🟠 عالي | إلغاء دين + إعادة المسدد |
| دفع زائد غير مسجل | عالي | منخفض | 🟡 متوسط | عرض الباقي + لا رصيد دائن |
