# آية موبايل - قرارات التصميم
## 10) Architecture Decision Records (ADRs)

---

## 📋 مقدمة

هذا المستند يوثق جميع القرارات التصميمية والتقنية المتخذة في مشروع "آية موبايل". كل قرار موثق مع السياق والبدائل والأسباب.

## ADR-018: ترتيب تسديد الديون

| الجانب | التفاصيل |
|--------|----------|
| **التاريخ** | 10 فبراير 2026 |
| **الحالة** | ✅ مُحدّث (22 فبراير 2026) |
| **السياق** | عند تسديد دين، أي قيد يُسدّد أولاً؟ |
| **القرار** | FIFO افتراضياً: الدفعة تُطبّق على الدين الأقدم (`MIN(due_date)`) حتى يصفر ثم التالي. أحمد (Admin) يستطيع تجاوز FIFO وتحديد دين معين يدوياً عبر `debt_entry_id` (V1) |
| **البدائل** | (1) لا ترتيب إجباري — مرفوض لأنه يسمح بتراكم ديون قديمة. (2) الأكبر أولاً — مرفوض لأنه لا يعكس الأولوية الزمنية |
| **السبب** | FIFO يضمن تصفية الديون الأقدم أولاً (أفضل ممارسة محاسبية)، مع إبقاء المرونة لأحمد عند الحاجة |
| **الأثر** | `create_debt_payment()`: إذا `debt_entry_id` غير محدد → FIFO تلقائي. إذا محدد → الدين المحدد. التوزيع يُسجل في `debt_payment_allocations`. واجهة التسديد تعرض الديون مرتبة بالأقدم أولاً مع إمكانية التحديد اليدوي لأحمد |

## ADR-041: سياسة ترقيم المستندات والفجوات (Document Numbering & Gaps Policy)

| الجانب | التفاصيل |
|--------|----------|
| **التاريخ** | 13 فبراير 2026 |
| **الحالة** | ✅ مُتخذ |
| **السياق** | الفواتير والمرتجعات وأوامر الشراء تحتاج أرقام تسلسلية. هل نسمح بفجوات في الترقيم؟ |
| **القرار** | نعم — الفجوات مقبولة. الأرقام تولّد من DB sequence. لا نضمن تسلسل بلا فجوات |

### صيغ ترقيم المستندات

| نوع المستند | الصيغة | مثال | المصدر |
|-------------|--------|------|--------|
| فاتورة | `AYA-YYYY-NNNNN` | `AYA-2026-00042` | DB sequence |
| مرتجع | `AYA-YYYY-NNNNN` | `AYA-2026-00005` | DB sequence |
| أمر شراء | `AYA-YYYY-NNNNN` | `AYA-2026-00012` | DB sequence |
| تسديد دين | `AYA-YYYY-NNNNN` | `AYA-2026-00003` | DB sequence |
| تسديد مورد | `AYA-YYYY-NNNNN` | `AYA-2026-00001` | DB sequence |
| تحويل | `AYA-YYYY-NNNNN` | `AYA-2026-00002` | DB sequence |
| صيانة | `AYA-YYYY-NNNNN` | `AYA-2026-00007` | DB sequence |

### سياسة الفجوات (Gaps)

| الحالة | هل تحدث فجوة؟ | مقبولة؟ |
|--------|-------------|---------|
| Transaction rollback | نعم — sequence تقدّم لكن لم يُحفظ | ✅ نعم |
| إلغاء فاتورة | لا — الرقم يبقى مع status=cancelled | ✅ نعم |
| idempotency duplicate | لا — نفس الرقم يُرجع | ✅ نعم |
| DB restart | نعم — sequence قد تقفز (حسب cache) | ✅ نعم |

> **المبدأ:** الفجوات في الترقيم ليست مشكلة. الأهم هو **عدم التكرار**. كل رقم فريد ولا يُعاد استخدامه.

### قواعد Idempotency Key

| القاعدة | التفاصيل |
|---------|---------|
| **المصدر** | Frontend يولّد `UUID v4` قبل إرسال الطلب |
| **النطاق** | Per-operation (كل عملية كتابة لها مفتاح) |
| **التحقق** | Backend يفحص `idempotency_key` في الجدول المستهدف |
| **السلوك عند التكرار** | يُرجع النتيجة السابقة (HTTP 409 + existing record) |
| **المدة** | لا expiry — المفتاح يبقى دائماً مع السجل |
| **الجداول التي تدعمه** | `invoices`, `returns`, `debt_entries`, `debt_payments`, `supplier_payments`, `topups`, `transfers`, `maintenance_jobs`, `purchase_orders`, `expenses` |
| **استثناء موثّق** | `daily_snapshots` لا يستخدم `idempotency_key`؛ السلطة عبر `UNIQUE(snapshot_date)` (Natural-Key Idempotency) |

| **القرار** | تطبيق نموذج Revoke-All-First: سحب جميع الصلاحيات أولاً (`REVOKE ALL`)، ثم إعادة منح `SELECT` فقط عبر RLS. جميع عمليات الكتابة تتم عبر `service_role` من طبقة API فقط |
| **البدائل** | الاستمرار بـ RLS مع INSERT — مرفوض لأن المستخدم يستطيع تجاوز الدوال والإدراج المباشر |
| **السبب** | مبدأ Least Privilege: لا أحد يملك صلاحية أكثر مما يحتاج. إغلاق مساحة الهجوم بالكامل |
| **الأثر** | تحديث جميع سياسات RLS. POS يرى البيانات فقط (SELECT). جميع INSERT/UPDATE/DELETE تتم عبر الدوال المُستدعاة بـ service_role |

## ADR-046: Authority Locks (Governance Hardening)

| الجانب | التفاصيل |
|--------|----------|
| **التاريخ** | 5 مارس 2026 |
| **الحالة** | ✅ مُتخذ |
| **السياق** | إغلاق تعارضات P0 ومنع Multiple Operational Truth قبل بدء البناء |
| **القرار** | اعتماد Locks إلزامية مرجعية قبل التنفيذ |

### Locks

| Lock | القاعدة | ما يُعد Regression | موضع الإنفاذ |
|------|---------|-------------------|--------------|
| `LOCK-NoBackdate` | `invoice_date`/`entry_date`/`snapshot_date` = `CURRENT_DATE` فقط | أي نص يسمح بتاريخ سابق | `01`, `04`, `05`, `13`, `15` |
| `LOCK-DriftAuthority` | فحص الانحراف عبر `fn_verify_balance_integrity()` فقط | أي اسم بديل للدالة/route/job | `13`, `15`, `22`, `25`, `03`, `05` |
| `LOCK-Suppliers-RLS` | POS بلا `SELECT` مباشر على `suppliers` | أي نص يمنح POS direct read | `05`, `13`, `18` |
| `LOCK-IdempotencyAuthority` | `create_debt_manual` يتطلب `idempotency_key`; `create_daily_snapshot` idempotent بـ `UNIQUE(snapshot_date)` | أي تعارض بين schema/contracts/errors | `05`, `13`, `15`, `16`, `25` |
| `LOCK-DeviceContract` | Web App يعمل على phone/tablet/laptop/desktop مع installability حسب دعم المتصفح | أي نص يحصر التشغيل بـ Chrome-only أو tablet/desktop فقط | `01`, `13`, `22`, `11`, `README` |
| `LOCK-SINGLE-BRANCH` | Aya Mobile يعمل بافتراض **Single-Branch** في MVP (لا `branch_id` ولا `tenant_id` في العقود التشغيلية) | أي إضافة branch/tenant scope أو مفاتيح مركبة مرتبطة بالفروع بدون ADR توسعة رسمي | `01`, `05`, `13`, `25`, `27`, `30` |

## ADR-047: طبقة الصلاحيات الدقيقة فوق الدور الأساسي

| الجانب | التفاصيل |
|--------|----------|
| **التاريخ** | 11 مارس 2026 |
| **الحالة** | ✅ مُتخذ |
| **السياق** | النظام الحالي يعتمد على `profiles.role` بقيمتين فقط (`admin`, `pos_staff`). توسعة V2 تحتاج صلاحيات تشغيلية أدق دون فتح authority جديدة على مستوى DB أو كسر Blind POS أو grants الحالية |
| **القرار** | الإبقاء على `profiles.role` كـ **coarse authority ceiling**، وإضافة طبقة ثانية عبر `permission_bundles` + `role_assignments` لتحديد القدرات التشغيلية داخل حدود الدور الأساسي |
| **البدائل** | (1) توسيع ENUM نفسه إلى عشرات الأدوار التشغيلية — مرفوض لأنه يخلط authority بالوظائف التشغيلية. (2) منح grants/RLS مختلفة لكل bundle — مرفوض لأنه يفتح shadow paths ويعقد الحوكمة |
| **السبب** | الحفاظ على نموذج authority الحالي مع السماح بتخصيصات V2 مثل الجرد، التقارير، الصيانة، والخصومات دون المساس بـ API-first أو Revoke-All-First |
| **الأثر** | `profiles.role` يبقى `admin` أو `pos_staff` فقط. أي bundle يجب أن يلتزم `base_role` الموافق له. التحقق من bundles يتم في API/server layer فقط. لا direct grants جديدة، ولا bundle يرفع `pos_staff` إلى صلاحيات DB من مستوى Admin |

### قواعد إلزامية

1. `profiles.role` يحدد السقف الأعلى، و`permission_bundles` يحدد التفاصيل التشغيلية داخل هذا السقف فقط.
2. لا يجوز لأي bundle أن يمنح `pos_staff` القدرة على تجاوز `fn_require_admin_actor()`.
3. لا يجوز فتح direct read/write path جديد بسبب bundles؛ كل التنفيذ يبقى عبر API المحمية.
4. تعيين bundles وإلغاؤها يجب أن يكون auditable عبر `role_assignments` و`audit_logs`.

## ADR-048: حوكمة الخصم بحسب bundle مع Audit إلزامي

| الجانب | التفاصيل |
|--------|----------|
| **التاريخ** | 11 مارس 2026 |
| **الحالة** | ✅ مُتخذ |
| **السياق** | الحد الحالي للخصم يعتمد فقط على `system_settings.max_pos_discount_percentage` مع استثناء Admin. توسعة الصلاحيات الدقيقة تحتاج ضبطًا أدق للخصم بحسب bundle، مع منع أي override صامت |
| **القرار** | ربط حدود الخصم مستقبلاً بـ `permission_bundles.max_discount_percentage` و`discount_requires_approval` مع بقاء `system_settings.max_pos_discount_percentage` كـ baseline عام. عند تجاوز الحد المسموح يجب إعادة `ERR_DISCOUNT_APPROVAL_REQUIRED` أو `ERR_DISCOUNT_EXCEEDED` حسب العقد |
| **البدائل** | (1) ترك الخصم global فقط — مرفوض لأنه لا يغطي تنوع الأدوار. (2) السماح بتجاوز صامت مع audit لاحق — مرفوض لأنه يخلق واقعًا تشغيليًا غير مضبوط |
| **السبب** | التحكم بالخصم من أكثر المسارات حساسية على الربحية والحَوْكمة التشغيلية. يجب أن يكون قرار الخصم قابلاً للتدقيق ومربوطًا بالدور/bundle لا بالشخص فقط |
| **الأثر** | `create_sale` و`edit_invoice` ستخضعان لاحقًا لفحص bundle cap + approval policy. أي approval أو override يجب أن يسجل في `audit_logs` ولا يفتح bypass جديدًا على مستوى DB |
