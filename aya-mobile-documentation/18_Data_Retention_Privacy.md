# آية موبايل - الاحتفاظ بالبيانات والخصوصية
## 18) Data Retention & Privacy Policy

---

## 📋 مقدمة

هذا المستند يوثق سياسة الاحتفاظ بالبيانات وحماية الخصوصية في نظام آية موبايل. النظام يعمل كتطبيق **داخلي** (Internal) لمتجر واحد ومستخدمين اثنين فقط. لا يتعامل مع بطاقات ائتمان، وقد يخزن رقم هوية وطني **اختياري** لعميل الدين عند الحاجة التشغيلية (مرئي للإدارة فقط). لا يحتاج امتثال GDPR.

---

## 🗄️ سياسة الاحتفاظ بالبيانات (Data Retention)

### المبدأ الأساسي

> **Soft Delete إلزامي (ADR-022):** لا يتم حذف أي سجل مالي أو تشغيلي فعلياً من قاعدة البيانات. يتم تعطيله عبر `is_active = false` أو `status = 'cancelled'`.

### فترات الاحتفاظ

| نوع البيانات | فترة الاحتفاظ | السبب |
|-------------|--------------|-------|
| **الفواتير** (`invoices`) | دائم | سلامة مالية — لا حذف |
| **القيود المحاسبية** (`ledger_entries`) | دائم | Append-Only (ADR-032) — لا تعديل ولا حذف |
| **سجل التدقيق** (`audit_logs`) | دائم | Append-Only (ADR-039) — لا تعديل ولا حذف |
| **المرتجعات** (`returns`) | دائم | مرتبطة بالفواتير |
| **الديون** (`debt_entries`, `debt_payments`) | دائم | تتبع مالي |
| **اللقطات اليومية** (`daily_snapshots`) | دائم | مرجع تاريخي |
| **المنتجات** | Soft Delete | `is_active = false` — تبقى للرجوع إليها في الفواتير السابقة |
| **الموردين** | Soft Delete | `is_active = false` |
| **عملاء الدين** | Soft Delete | `is_active = false` |
| **الإشعارات** (`notifications`) | 90 يوم | يمكن حذف الإشعارات المقروءة بعد 90 يوم (Scheduled Job اختياري) |

## 🔒 حماية البيانات

### البيانات الشخصية المُخزّنة

| البيانات | الجدول | الحساسية | المعالجة |
|----------|--------|---------|----------|
| اسم الموظف | `profiles` | منخفضة | مرئي لـ Admin فقط |
| بريد الموظف | `profiles` | منخفضة | يُستخدم لتسجيل الدخول فقط |
| اسم عميل الدين | `debt_customers` | منخفضة | مرئي لـ Admin + POS |
| هاتف عميل الدين | `debt_customers` | منخفضة | لإرسال إشعارات واتساب (V2) |
| رقم هوية عميل الدين (اختياري) | `debt_customers.national_id` | متوسطة | مرئي لـ Admin فقط — POS لا يراه |
| اسم عميل الصيانة | `maintenance_jobs` | منخفضة | للتواصل فقط |
| هاتف عميل الصيانة | `maintenance_jobs` | منخفضة | للتواصل فقط |
| عنوان IP | `audit_logs` | منخفضة | للتدقيق الأمني فقط |

### ما لا يُخزّن

| البيانات | السبب |
|----------|-------|
| أرقام بطاقات الائتمان | لا يتعامل النظام مع الدفع الإلكتروني المباشر |
| صور/نسخ الهوية الوطنية | غير مطلوبة |
| كلمات المرور (نص واضح) | تُدار عبر Supabase Auth (hashed) |
| بيانات الموقع الجغرافي | غير مطلوبة |
| أرقام IMEI | قرار تصميمي — لا تتبع (ADR-010) |

## 🛡️ ضوابط الوصول

| المستخدم | ما يراه | ما لا يراه |
|---------|---------|-----------|
| **Admin** | كل شيء | — |
| **POS Staff** | فواتيره + بيانات تشغيل محدودة عبر Views آمنة (`v_pos_products`, `v_pos_accounts`, `v_pos_debt_customers`) | أسعار التكلفة، الأرصدة الفعلية، حدود الائتمان، الأرباح، `ledger_entries`، سجل التدقيق، اللقطات اليومية، الإعدادات |
| **مستخدم غير مُسجّل** | لا شيء — النظام يتطلب تسجيل دخول | كل شيء |

> **مبدأ إلزامي:** Blind POS يُطبّق على مستوى البيانات (RLS + Views آمنة) وليس على مستوى الواجهة فقط.

**تطبيق مباشر على RLS:**  
POS لا يملك `SELECT` مباشر على الجداول المالية الحساسة (`ledger_entries`, `suppliers`, `purchase_orders`, `purchase_items`, `supplier_payments`, `topups`, `transfers`, `expenses`, `reconciliation_entries`)؛ الوصول يكون عبر API محمي عند الحاجة التشغيلية فقط.

### Suppliers RLS Contract (Authoritative)

| الدور | Direct SELECT على `suppliers` | النطاق | الحقول المسموحة |
|------|-------------------------------|--------|------------------|
| Admin | لا (على جدول `suppliers` مباشرة) | كامل المتجر (MVP: فرع واحد) عبر `admin_suppliers` View أو API | كل الحقول عبر View/API |
| POS Staff | لا | — | لا يوجد direct table read |

**عبر API فقط للـ POS (عند الحاجة التشغيلية):**
- الحقول المسموحة: `id`, `name`, `is_active`.
- الحقول المحجوبة: `current_balance`, `phone`, `address`.
- هذا جزء إلزامي من Blind POS (بيانات التكلفة/الالتزامات لا تظهر للـ POS).

## 🗑️ حذف البيانات

### لا يُسمح بحذف:
- أي سجل في `ledger_entries` (ADR-032)
- أي سجل في `audit_logs` (ADR-039)
- أي فاتورة — تُلغى فقط (`status = 'cancelled'`)

### يُسمح (عبر Admin فقط):
- تعطيل منتج: `products.is_active = false`
- تعطيل مورد: `suppliers.is_active = false`
- تعطيل عميل دين: `debt_customers.is_active = false`
- حذف إشعارات مقروءة قديمة (> 90 يوم)

## 🧭 Addendum مخطط لما بعد `PX-07` (V2 Privacy Controls)

> هذا القسم يضيف حدود الخصوصية المطلوبة قبل تنفيذ `PX-08 .. PX-14`.
> هو **عقد تخطيطي معتمد** وليس ادعاء بأن هذه features مفعلة الآن.

### 1) روابط الإيصالات العامة (`receipt links`)

| البند | السياسة |
|-------|---------|
| نوع الرابط | token opaque غير قابل للتخمين |
| الصلاحية | ينتهي بعد مدة قابلة للضبط، والافتراضي `7` أيام |
| الإلغاء | Admin يمكنه revoke token في أي وقت |
| ما يُعرض | رقم الفاتورة، التاريخ، العناصر، الإجمالي، اسم المتجر، بيانات مشاركة أساسية |
| ما لا يُعرض | `cost_price`, `profit`, `avg_cost_price`, `internal_notes`, `audit data`, `current_balance`, أي حقول إدارية |
| الاستخدام | read-only فقط، بدون أي عمليات كتابة |

### 2) رسائل واتساب وdelivery logs

| البند | السياسة |
|-------|---------|
| الغرض | receipt sharing + debt reminders فقط |
| أقل قدر من البيانات | تخزين `template_key`, `provider_message_id`, `status`, `reference_type/id`, ووقت الإرسال |
| رقم الهاتف في الواجهة | masking جزئي كلما أمكن |
| body الكامل للرسالة | لا يُخزّن افتراضيًا إلا إذا وُجدت ضرورة تشغيلية واضحة ومحددة |
| الإخفاقات | `last_error` مسموح بشكل sanitized فقط |
| الحذف/الاحتفاظ | يراجع مع retention الإشعارات وسجل التشغيل، ولا يتحول إلى أرشيف رسائل دائم بلا حاجة |

### 3) الحزم المحمولة (`export/import/backup`)

| البند | السياسة |
|-------|---------|
| الصلاحية | Admin فقط |
| التتبع | كل export/import/restore يجب أن يُسجل في `audit_logs` |
| الروابط/الملفات | bounded + expirable + قابلة للإبطال عند الإمكان |
| الاستعادة | restore drill فقط على بيئة معزولة |
| الحقول الحساسة | تراجع قبل التوليد حسب نوع الحزمة (`PII`, `national_id`, phones) |
| الهدف | portability تشغيلية لا نسخة مفتوحة بلا ضوابط |

**تفصيل إلزامي للحزم المحمولة:**
- download لأي package يبقى `Admin-only` ولا يتحول إلى public link.
- حزم `customers` يجب أن تعرض `phone_masked` فقط وتحذف `national_id` و`phone` الخام.
- حزم `backup` لا تتضمن بيانات العملاء الدائنة الخام ما لم يوجد عقد مستقل يفرض ذلك لاحقًا.
- revoke لأي package يجب أن يمنع التنزيل مباشرة ويحوّل الحالة إلى `revoked/expired`.

### 4) الإشعارات ومراكز التنبيه

| البند | السياسة |
|-------|---------|
| Admin | يرى كل الإشعارات التشغيلية |
| POS Staff | يرى إشعاراته أو الإشعارات scoped لدوره فقط |
| التنبيهات المالية الحساسة | لا تظهر للـ POS إلا إذا كانت جزءًا من دوره التشغيلي الصريح |
| الحذف | يظل مرتبطًا بسياسة `90 يوم` الحالية للإشعارات المقروءة ما لم يثبت احتياج آخر |

### 5) مبدأ خصوصية V2

> أي feature جديدة في `V2` تفتح سطح وصول جديد (`public link`, `message delivery`, `export package`, `advanced alert`) يجب أن تبدأ من قاعدة:
> **أقل قدر من البيانات + صلاحية محددة + سجل تدقيق + قابلية إبطال**.
